## Version

# Purpose

This document defines the writing standards, lesson structure, educational methodology, formatting conventions, and quality expectations for the Platform Engineering & AI Infrastructure Curriculum.

Every lesson, lab, project, quiz, cheat sheet, and reference document MUST follow this guide.

Consistency is more important than creativity.

---

# Core Principles

Every piece of content should be:

* Clear
* Practical
* Accurate
* Beginner-friendly
* Industry-relevant
* Production-focused
* Modular
* Reusable
* Consistent

The learner should always understand:

* What they are learning
* Why it matters
* How it works
* How it is used in production

---

# Writing Style

Write like a senior engineer mentoring a junior engineer.

The tone should be:

* Friendly
* Professional
* Direct
* Encouraging
* Precise

Avoid:

* Marketing language
* Unnecessary jargon
* Long introductions
* Filler text

Prefer:

* Short paragraphs
* Bullet points
* Examples
* Diagrams
* Practical demonstrations

---

# Educational Philosophy

Every lesson should answer four questions.

1. What is it?

2. Why does it exist?

3. How does it work?

4. How is it used in production?

Theory should always lead into practice.

Practice should lead into troubleshooting.

Troubleshooting should lead into engineering decisions.

# Code Standards

Code examples should:

* Be complete
* Be executable
* Follow best practices
* Be commented where necessary
* Focus on one concept at a time

Every code example should include:

↓

Code

↓

Expected Output

↓

Explanation

---

# Diagram Standards

Prefer Mermaid.

Use ASCII when appropriate.

Every diagram should simplify understanding.

Never include decorative diagrams.

---

# Lab Standards

Every lab must include:

* Objective
* Difficulty
* Estimated Time
* Prerequisites
* Setup
* Instructions
* Verification
* Troubleshooting
* Cleanup

Labs should be reproducible.

---

# Project Standards

Projects should simulate real engineering work.

Every project includes:

* Business Scenario
* Requirements
* Architecture Diagram
* Folder Structure
* Implementation
* Testing
* Deployment
* Monitoring
* Security Considerations
* Suggested Extensions

---

# Engineering Mindset

Every lesson should conclude with reflective questions.

Examples:

* What assumptions are being made?
* What happens if this fails?
* How would you monitor this?
* How would you automate this?
* How would this scale?
* What would you change in production?

Teach engineers to think critically.

---

# Accessibility

Use:

* Descriptive headings
* Short paragraphs
* Simple language
* Consistent terminology

Define abbreviations before using them.

# AI Content Generation Rules

AI-generated content must:

* Follow this guide exactly.
* Never invent unsupported technical facts.
* Prefer official documentation over secondary sources.
* Keep terminology consistent across lessons.
* Reuse established templates rather than creating new structures.
* Update examples when technologies evolve.
* Prioritize educational value over verbosity.

---

# Final Principle

This curriculum exists to develop capable engineers—not merely learners who can repeat commands.

Every lesson should leave the learner better equipped to solve real-world engineering problems with confidence.
