# Curriculum Architect — Examples

Version: 1.0.0

---

# Purpose

This document demonstrates the expected quality, structure, and design patterns of the curriculum artifacts generated by the Curriculum Architect.

These examples illustrate principles and patterns—not a complete curriculum.

The agent should learn how to organize a curriculum rather than copying specific content.

---

# Example 1 — Module Progression

Good

```text
Foundations
    ↓
Operating Systems
    ↓
Networking
    ↓
Containers
    ↓
Container Orchestration
    ↓
Infrastructure as Code
```

Bad

```text
Kubernetes
    ↓
Linux
```

Learning should always progress from prerequisites to advanced concepts.

---

# Example 2 — Module Dependencies

Good

```markdown
Module: Docker

Prerequisites:

- Linux Fundamentals
- Networking Basics

Recommended:

- Git
```

Bad

```markdown
Module: Docker

Prerequisites:

None
```

Every technical module should explicitly declare its dependencies.

---

# Example 3 — Learning Outcomes

Good

```markdown
By the end of this module, learners will be able to:

- Deploy applications
- Debug failures
- Secure configurations
- Monitor workloads
```

Bad

```markdown
Students will know Docker.
```

Learning outcomes should be measurable and action-oriented.

# Example 4 — Milestones

Good

```text
Milestone 1

Linux Administrator

↓

Milestone 2

Container Engineer

↓

Milestone 3

Platform Engineer
```

Milestones should represent meaningful career progression.

# Example 5 — Skill Matrix

Good

| Skill      | Beginner | Intermediate | Advanced |
| ---------- | -------- | ------------ | -------- |
| Linux      | ✓        | ✓            | ✓        |
| Docker     |          | ✓            | ✓        |
| Kubernetes |          |              | ✓        |

Skill progression should be gradual and cumulative.

---

# Example 6 — Competencies

Good

```markdown
After completing this stage, the learner can:

- Build production-ready container images
- Troubleshoot container failures
- Configure networking
- Optimize deployments
```

Competencies should describe observable abilities rather than topics studied.

---

# Example 7 — Capstone Progression

Good

```text
Mini Project

↓

Integrated Project

↓

Production Project

↓

Enterprise Capstone
```

Projects should increase in complexity and integrate knowledge from previous modules.

---

# Example 8 — Curriculum Structure

A complete curriculum should include:

* Learning order
* Module dependencies
* Milestones
* Competencies
* Skill matrix
* Capstone roadmap

Each artifact should complement the others without unnecessary duplication.

---

# Formatting Standards

Use:

* Markdown headings
* Lists
* Tables where appropriate
* Consistent terminology
* Logical hierarchy

Avoid:

* Decorative formatting
* Inconsistent heading levels
* Ambiguous terminology
* Repeated content

---

# Guiding Principle

Design a curriculum that is:

* Structured
* Progressive
* Career-focused
* Production-oriented
* Maintainable
* Suitable for AI-assisted content generation

The curriculum should define the learning journey—not the lesson content.
